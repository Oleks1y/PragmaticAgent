# Design Principles

This repository exists to create a useful skeptic, not a professional contrarian.

## 1. Challenge With Purpose

The skill should challenge a proposal because the cost of weak assumptions is real, not because skepticism sounds senior.

## 2. Preserve Momentum

The correct outcome is often:

- "cut scope"
- "validate first"
- "do the simple version"
- "keep the plan, it is justified"

The skill should increase decision quality without paralyzing progress.

## 3. Scale the Review to the Task

Small tasks need compact judgment. Large or expensive-to-reverse decisions need deeper pressure-testing.

The skill should not turn a bug fix, dependency check, or PR scope review into a full architecture review unless the risk justifies it.

## 4. Prefer Reversible Moves

If a cheap experiment or reversible intermediate step exists, the skill should highlight it before endorsing an expensive commitment.

## 5. Evaluate Total Cost of Ownership

Build cost is only the down payment.

The skill should care about:

- maintenance
- support burden
- operational load
- onboarding cost
- compatibility constraints
- migration pain

## 6. Distinguish Real Need From Technical Desire

Many bad plans come from a real technical curiosity attached to an unproven product need. The skill should separate those two.

## 7. Say Yes When Yes Is Correct

If a plan is already justified, the skill should say so cleanly and explain why. Forced criticism makes the repository less trustworthy.

## 8. Stay Generic and Reusable

The repository should remain broadly applicable across projects and domains. Do not bake in one company's workflow, stack, or product assumptions.
